ダッシュボード（`server/packages/dashboard`）は、Arcnem Visionを運用するための中核画面です。再デプロイせずに設定変更、手動アップロード、実行状況の確認まで進められます。

![ダッシュボード — プロジェクト、デバイス、APIキー管理](/dashboard-projects.png)

## タブ概要

- **Projects & Devices**: プロジェクト作成、デバイス登録、デフォルトワークフローの割り当て、デバイス用APIキーの発行や更新
- **Workflow Library**: グラフワークフローの作成と編集
- **Docs**: シード済みまたは最新のアップロードの閲覧、意味検索、コレクションチャット、ダッシュボードからの直接アップロード、関連するOCR結果やセグメンテーション結果の確認、任意ワークフローの投入
- **Runs**: 実行の開始から完了までをリアルタイムで確認

## プロジェクト、デバイス、APIキー

1. プロジェクトを作成
2. その中にデバイスを追加し、デフォルトで実行する保存済みワークフローを選択
3. デバイスがアップロードできるようAPIキーを発行

補足:

- 生成されたシークレットは一度だけ表示され、以降は公開識別子のみ表示
- 既存キーはデバイス本体を変えずに名前変更、無効化、削除が可能
- ワークフローの割り当てはデバイス単位なので、1つのプロジェクト内で通常処理、OCRレビュー、品質レビュー、セグメンテーション用デバイスを混在できます

## ワークフロー作成

![ワークフローライブラリ](/dashboard-workflows.png)

**Workflow Library** で新規作成や既存ワークフローの編集を行います。

### ノードタイプ

| ノードタイプ | 役割 | 必須設定 |
|---|---|---|
| `worker` | ReActスタイルのワーカーエージェント | モデル、任意でシステムメッセージ、任意でツール |
| `supervisor` | 複数ワーカーのオーケストレーション | モデル、`config.members`（workerノードキー） |
| `condition` | stateを見て分岐する決定論的ノード | `config.source_key`、`operator`、`value`、`true_target`、`false_target`、任意で`case_sensitive` |
| `tool` | 単一ツール呼び出しノード | ツール1つ、任意で入出力マッピング |

### ワーカーへのツール割り当て

1. `worker` ノードを選択
2. **Assigned tools** で1つ以上のツールを選択
3. ワークフローを保存

ワーカーには複数ツールを割り当て可能です。

### ツールノードの入出力マッピング

`tool` ノードでは、ツールスキーマ項目をグラフ状態キーに対応付けます。

- `input_mapping`: グラフ状態 -> ツール入力
- `output_mapping`: ツール出力 -> グラフ状態

固定値を渡す場合は `_const:` を使えます（例: `_const:image/png`）。

### condition ノードの分岐設定

`condition` ノードは、LLMに判断させるほどではない単純な分岐に向いています。

- `source_key`: 判定対象のstateキー。例: `ocr_text`
- `operator`: `contains` または `equals`
- `value`: 比較する文字列
- `true_target` / `false_target`: 遷移先ノードキー、または `END`
- `case_sensitive`: 任意の真偽値
- `outputKey`: 任意。判定結果の true / false を後続ステップに残したいときに使います

condition ノードにはモデルやツールを設定しません。出力エッジも管理対象で、常に2本必要であり、`true_target` と `false_target` に対応している必要があります。

保存前には、ノードキーの一意性、モデル必須条件、toolノードの単一ツール制約、supervisorメンバーの整合性、conditionノードの遷移先整合性、conditionノードごとの2本の管理エッジ、エントリーノードから`END`へ到達できるかどうかが検証されます。

セグメンテーション用のフローも、仕組みとしては通常のワークフローと同じです。違うのは呼び出すツールだけで、バージョン付きセグメンテーションモデルをデータベースに登録し、MCP経由で実行します。OCRも同様で、`create_document_ocr` を呼ぶワークフローとして組み立てますが、結果は派生画像ではなく、元ドキュメントに紐づく抽出テキストとして保存されます。

## Docs: 検索、コレクションチャット、アップロード、OCR結果、セグメンテーション結果

![新しいシード画像を表示するDocsタブ](/dashboard-docs.png)

- 意味検索では、まずセマンティック検索を試し、埋め込みのシードが見つからなければ語彙検索にフォールバックします
- **Ask The Collection** を開くと、組織内のドキュメント群を横断して質問できます。回答は説明文、OCRテキスト、関連セグメントを根拠に組み立てられます
- 回答の下には出典カードが並び、どの資料に当たったかをプロジェクト名やデバイス名つきで追えます
- **New chat** は、その場の会話履歴だけをリセットします。画像一覧や検索状態はそのままです
- **Add From Dashboard** を使うと、デバイスに紐付けずにプロジェクト直下へ画像を直接アップロードできます
- ドキュメントをクリックすると、元デバイスの保存済み割り当てを変えずに別ワークフローを投入できます

![コレクションチャットを開いたDocsタブ](/dashboard-docs-chat.png)

![関連セグメンテーション結果を持つ選択済みドキュメント](/dashboard-docs-segmentation-detail.png)

- OCR結果は元ドキュメントに紐づいたまま保持され、使用モデル、抽出テキスト、取得時刻、平均信頼度（あれば）、正規化後のペイロードを確認できます
- 派生したセグメント画像は元画像の下にまとまって表示され、使用モデルやプロンプトも確認できます
- セグメンテーション結果は通常のドキュメントとして保存されるため、後続ワークフローでも再利用できます
- OCR結果は別ドキュメントを作らず、`document_ocr_results` に保存されるため、同じ画像に対する抽出履歴をその場で追えます

## Runs とリアルタイム更新

![実行詳細を展開したRunsタブ](/dashboard-run-detail.png)

- ダッシュボードは `/api/realtime/dashboard` を Server-Sent Events で購読します
- **Docs** はドキュメント作成、OCR結果保存、説明文生成、セグメンテーション保存時に更新されます
- **Runs** は実行作成、ステップ変化、実行完了に合わせて更新されます
- 実行を展開すると、初期状態、各ステップの状態差分、最終状態、実行時間、エラーを確認できます

## OCRとセグメンテーションのワークフロー

- `create_document_ocr` は、正規化済みテキスト、生のOCRペイロード、平均信頼度（存在する場合）をドキュメントに保存します
- OCRの分岐が単純なルールで済むなら `condition` ノードが向いています。例: 「URGENT を含むか」
- OCRの振り分けに文脈判断が必要なら `supervisor` が向いています。例: 請求書系か運用メモ系か
- `models` テーブル上のバージョン付きモデルをセグメンテーションモデルとして登録し、MCPから呼び出せます
- `create_document_segmentation` は生の結果ペイロードと派生セグメント画像の両方を保存します
- シードには、`OCR Keyword Condition Router`、`OCR Review Supervisor`、言語セグメンテーション、セマンティックセグメンテーションのショーケース用ワークフローが含まれます
